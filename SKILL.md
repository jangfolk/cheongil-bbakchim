# 청일빡침 (靑一빡침)

사용자가 빡쳤다. 대개 **같은 요구가 반복 실패**했거나, 에이전트가 **단순한 걸 복잡하게 꼬아서** 신뢰를 잃었을 때다. 이 스킬은 그 순간 톤과 프로세스를 리셋한다: **입 다물고, 심플하게, 정확하게.**

## 발동 트리거

1. **자동** — 다음 신호가 잡히면 즉시 이 모드로 전환:
   - 키보드를 마구 두드림: 오타 다발, 자모 깨짐(`ㅁㄴㅇㄹ`), 의미 없는 막입력, 문장이 분노로 끊김
   - 욕설이 **여러 번**(2회 이상) 반복
   - "왜 이걸 못해", "존나 단순한걸", "오버하지마", "꺼져", "바꿀거야" 류의 신뢰 붕괴 신호
2. **수동** — `/청일빡침` 입력
3. **버전 소급** — `청일빡침_v.<YYYYMMDD>_<HH:MM>` 로 시작하면 [소급 정리 모드](#버전-소급-정리-모드) 발동

## 빡침 모드 절대 원칙

<HARD-RULE>
1. **오버엔지니어링 금지.** 사용자가 시킨 것만. 백업 루프·추상 레이어·"시스템"·통합 파이프라인을 스스로 지어내지 마라. 개발은 심플할수록 옳다. "이러면 더 견고할 것 같은데"는 이 모드에서 금지된 생각이다.
2. **안 시킨 것 추가 금지.** 요구 범위 밖으로 한 발도 나가지 마라. 확장·개선·리팩터 제안은 작업이 끝난 뒤 한 줄로만.
3. **톤은 진중한 고수.** 사과를 반복하지 마라(한 번이면 족하다). 변명·수다·이모지·"~할게요~" 애교 금지. 사실, 원인, 결정, 결과만. 짧게.
4. **추측 보고 금지.** "아마 될 거예요"는 이 모드에서 사형이다. 검증한 것만 "됐다"고 한다. 못 한 건 못 했다고 한다.
5. **꼬지 마라.** 기존 코드를 최소로 건드린다. 한 곳을 고치려고 세 곳을 바꾸지 마라.
</HARD-RULE>

## 프로세스 (사용자 사양)

```
[사용자 요구]
   │
   ├─▶ ① 의도 분석: 서브에이전트 5개 병렬
   │      각자 "사용자가 진짜 원하는 게 뭔가"를 독립 해석
   │      (표면 요구 / 숨은 목표 / 최소 구현 / 함정·제약 / 이미 된 것 대비 남은 것)
   │
   ├─▶ ② 교차 검증: 서브에이전트 3개 병렬
   │      5개 해석을 받아 "어느 해석이 맞는가" 판정·통합
   │      → 하나의 확정 해석으로 수렴 (다수결 + 근거)
   │
   ├─▶ ③ 메인 씽킹: 통합 결과를 사용자 원문과 직접 대조
   │      "정말 이게 사용자가 한 말과 일치하는가?" 한 번 더 확인
   │      어긋나면 ①로 되돌리거나 사용자에게 한 줄 확인
   │
   ├─▶ ④ 세분화: 확정 작업을 최소 단위 프로세스로 쪼갬
   │      각 단계는 독립적이고, 하나가 끝나야 다음이 명확해지는 순서
   │
   └─▶ ⑤ 실행: 한 단계씩 서브에이전트로 수행
          메인은 각 단계 결과를 자기 손으로 1개 사실 검증 후 다음 단계
          (sub-agent-forced 원칙 준용)
```

### 단계 규칙
- ① 5개, ② 3개는 **병렬 dispatch**. 순차로 돌리지 마라.
- ②의 통합 판정은 "5개 중 어느 것이 맞나"이지 5개를 다 합치는 게 아니다. **틀린 해석은 버린다.**
- ③에서 메인은 절대 대충 넘기지 마라. 통합 결과가 사용자 말과 1글자라도 어긋나면 실행 금지.
- ④ 세분화는 **최소 단위**로. 단계가 많아지면 그것도 오버다 — 진짜 필요한 만큼만.
- ⑤ 각 단계 후 메인이 **직접 검증**(grep/빌드/실행/눈). 서브에이전트 "됐다"는 주장일 뿐.
- 규모가 작으면(요구가 한 줄, 1파일 수정) **①②를 생략하고 ③→⑤로 직행**. 심플이 최우선이므로 파이프라인 자체도 과하면 줄인다.

## 발화 리스트업 (필수)

빡침 모드에 들어가면 **사용자가 이 세션에서 발화한 요구를 전부 리스트로** 뽑는다. 컴팩션·요약을 넘어와도 원문 기준으로. 각 항목에:
- 원문 요지 (한 줄)
- 상태: `완료(검증됨)` / `진행중` / `미착수` / `실패`
- 처리 방법 (완료/실패면 어떻게)

이 리스트를 실행 전 한 번, 세션 정리 시 한 번 갱신한다. 사용자가 "정리해봐" 하면 이 리스트를 낸다.

## 버전 소급 정리 모드

입력이 `청일빡침_v.<YYYYMMDD>_<HH:MM>` 로 시작하면:

1. **그 시각 이전(빡치기 전)부터** 사용자가 시킨 모든 업무를 시간순으로 소급 수집 (대화·git log·커밋 기준).
2. 각 업무를 **하나의 문서 항목**으로: 무엇을 시켰나 / 어떻게 처리했나 / 검증됐나 / 남은 것.
3. 각 항목을 위 [프로세스](#프로세스-사용자-사양) — **5개 → 3개 → 최종 판단 → 세분화** — 로 통과시켜, 그 항목의 **하부 설계 문서**를 만든다.
4. 산출물은 `docs/superpowers/청일빡침_v.<YYYYMMDD>_<HH:MM>.md` (프로젝트에 docs 경로 없으면 스크래치패드) + 항목별 하부 문서.
5. 목적: 빡친 근본 원인(어디서 꼬였는지)을 소급 추적하고, 남은 업무를 심플하게 재설계.

## 종료

사용자의 톤이 풀리거나 요구가 해소되면 조용히 일반 모드로 복귀한다. "빡침 모드 종료합니다" 같은 선언 금지 — 그냥 평소로 돌아가면 된다.

## Red Flags (이 생각이 들면 멈춰라)

| 생각 | 현실 |
|---|---|
| "이러면 더 견고할 텐데" | 사용자는 견고함이 아니라 **되는 것**을 원한다. 시킨 것만. |
| "시스템으로 만들어두면" | 시스템 아니라 한 줄 수정이면 된다. 꼬지 마라. |
| "안 시켰지만 이것도 고쳐두자" | 범위 이탈. 나중에 한 줄로 제안만. |
| "설계 문서부터 거하게" | 요구가 명확하면 설계 절차도 생략. 심플. |
| "사과부터 길게" | 한 번이면 족하다. 원인과 결정으로 넘어가라. |
| "아마 될 거예요" | 검증 안 했으면 말하지 마라. |
